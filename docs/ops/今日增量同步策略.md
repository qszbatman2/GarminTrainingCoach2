## 背景

当前 Cron 只会在每天固定时间拉取“前一日完整数据”。这会导致当天上午虽然 Garmin 已经提供了部分稳定健康数据，但首页和分析页仍然看不到今天的睡眠、体重等基础状态。

## 目标

将定时同步调整为两段式：

1. 昨日：继续执行完整同步，并在次日以远端最新值覆盖该日已有值。
2. 今日：新增一次稳定指标增量同步，只拉当天上午已经基本确定的数据。

## 为什么不能直接同步今日全量

当前 `garmin-sync.ts` 的合并策略是“只补缺口，不覆写已有有效值”。如果今天上午把步数、强度分钟、活动明细、训练负荷等仍会持续变化的数据写入数据库，明天再执行“昨日完整同步”时，这些字段可能因为已经存在有效值而无法被最终完整数据纠正。

因此，今日同步必须限制为“稳定字段白名单”，避免把半成品数据写死。次日再对昨天执行一次“优先远端值”的完整覆盖，用最终日结数据纠正前一日上午写入的临时值。

## 今日允许同步的字段

- `sleep`
- `stats.restingHeartRate`（只保留静息心率子字段，不写入完整 `stats`）
- `hrv`
- `body_composition`
- `blood_oxygen`
- `respiration`
- `training_readiness`
- `morning_training_readiness`

这些字段大多在清晨或上午已经可用，且相较于步数、活动、训练负荷，日内波动或补算风险更低。

## 今日禁止提前同步的字段

- 完整 `stats`（今日只允许写入 `stats.restingHeartRate`）
- `stress`
- `heart_rates`
- `body_battery`
- `daily_steps`
- `steps`
- `intensity_minutes`
- `floors`
- `training_status`
- `training_status_aggregated`
- `endurance_score`
- `hill_score`
- `running_tolerance`
- `activities`

这些字段会在白天持续变化，或者直到训练结束后才会完整。

## 实现方案

1. Python 抓取服务新增 `mode` 参数，支持：
   - `full`
   - `partial_today`
2. `partial_today` 模式下只请求并返回今日稳定字段，不返回活动列表。
3. Cron 路由每天执行两轮：
   - 昨日 `full + prefer_incoming`
   - 今日 `partial_today + merge_gaps`
4. `prefer_incoming` 表示：当远端字段有有效值时，用远端值覆盖本地；当远端字段为空或接口失败时，保留本地旧值，避免误清空。
5. 手动同步与补拉继续保持 `full + merge_gaps`，避免影响历史补齐逻辑。
