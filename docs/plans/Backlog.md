# 私人 AI 教练发布 Backlog

## 1. 目标

面向“可发布体验”来规划后续功能，核心目标不是继续堆仪表盘，而是把当前 Garmin 数据平台升级成一个用户愿意每天打开的私人 AI 教练。

本 Backlog 只保留两类内容：

- 直接提升用户可感知体验的功能
- 不做就不适合发布的稳定性/可信度门槛

不纳入本次发布核心范围的内容：

- 过重的多 Agent 架构探索
- 纯后台运维能力扩张
- 只有技术美感、没有用户感知的重构

## 2. 发布目标定义

达到“可发布体验”至少应满足以下结果：

1. 用户完成绑定 Garmin 后，能稳定看到当天身体状态结论
2. 系统不只回答“能不能练”，还要回答“今天练什么、练多少、练到什么强度”
3. 系统能给出未来 7 天的基础训练安排，而不是只有单日建议
4. 当数据缺失或可信度不足时，系统会明确提示，不装懂
5. 页面表达足够简单，用户一眼就能做决定，符合“一卡一决策”

## 3. 当前产品基础

当前项目已经具备以下可复用基础：

- Garmin 账号绑定、单日同步、补拉、对账主链路
- 日级健康数据和活动数据入库
- 睡眠、HRV、静息心率、压力、Body Battery、训练负荷等指标提取
- 基于规则引擎的“可训 / 慎训 / 不训”判断
- AI 对规则结果做自然语言解释
- 数据分析页与同步排查页

结论：现在已经有“分析器”雏形，但还没有形成真正的“教练闭环”。

## 4. 优先级规则

- `P0`：上线前必须完成，否则产品体验不成立或不适合发布
- `P1`：首发后 1-2 个版本应补齐，决定产品是否具备明显竞争力
- `P2`：增强项，能拉开差异，但不阻塞首发

## 6. 推荐发布范围

建议首发版本只承诺一个清晰定位：

**“基于 Garmin 数据的每日训练决策 + 7 天基础训练计划 AI 教练。”**

首发范围建议包含以下 8 个功能：

1. 今日身体状态卡
2. 今日训练处方
3. 每周训练负荷评估
4. 7 天基础训练计划
5. 数据可信度提示
6. 新用户首日闭环
7. 历史分析可回看
8. Garmin 凭证安全改造

原因：

- 这 8 项已经能形成完整用户价值闭环
- 再往上加专项模式、动态调计划，会拉长工期并显著增加规则复杂度
- 首发先把“判断准 + 建议清 + 页面稳”做扎实，比堆更多功能更重要

## 7. P0 详细拆分

### P0-1 今日身体状态卡

**目标**

让首页第一屏直接完成“今天能不能练”的决策。

**最低交付要求**

- 结论明确：可训 / 慎训 / 休息
- 风险等级明确：低 / 中 / 高
- 至少展示 3 个关键证据：如 HRV、睡眠、静息心率、负荷比
- 证据必须是“相对个人基线”的解释，而不是只报绝对值

### P0-2 今日训练处方

**目标**

让建议从“判断”升级为“处方”。

**最低交付要求**

- `recommendedWorkoutType`：恢复 / 轻松有氧 / 阈值 / 长课 / 休息
- `durationRange`：建议时长区间
- `intensityRange`：建议强度区间
- `whyToday`：为什么今天这样练
- `whatNotToDo`：今天不该做什么

### P0-3 每周训练负荷评估

**目标**

让用户知道当前周节奏是否合理，避免只看单日。

**最低交付要求**

- 本周训练量结论
- 本周训练强度结论
- 与最近 4 周同进度对比
- 针对下周的调整建议

### P0-4 7 天基础训练计划

**目标**

让产品具备最基本的“教练规划能力”。

**最低交付要求**

- 未来 7 天逐日安排
- 每天至少包含：训练类型、训练目的、建议时长、强度提示
- 自动插入恢复日，避免连续高强度
- 与当日 readiness 结果一致，不能自相矛盾

### P0-5 数据可信度提示

**目标**

用户必须知道系统是在“有把握地判断”还是“基于残缺数据保守判断”。

**最低交付要求**

- 标记数据完整度
- 标记核心缺失字段
- 对 AI 文案做收敛，不允许在低可信度下给强结论

### P0-6 新用户首日闭环

**目标**

确保第一次进入的用户不会卡在“有系统但没价值”。

**最低交付要求**

- 登录/注册
- Garmin 绑定
- 完成首次同步
- 自动生成首份分析
- 自动展示今日处方和 7 天计划

### P0-7 历史分析可回看

**目标**

建立连续感和可追溯性。

**最低交付要求**

- 支持查看最近 14-30 天的每日结论
- 可看到结论变化趋势
- 可看到建议是否因恢复状态而调整

### P0-8 Garmin 凭证安全改造

**目标**

补齐对外发布的最低安全门槛。

**最低交付要求**

- Garmin 凭证不以明文形式直接持久化
- 失败和重试链路不泄露敏感信息
- 绑定与更新操作有明确反馈

## 8. P1 版本建议

P1 适合在首发后 1-2 个版本推进，重点不是补页面，而是拉开“私人教练”差异：

1. 个体化恢复评分模型
2. 自研训练负荷模型
3. 动态调计划
4. 专项目标模式
5. 单次训练复盘
6. 每周复盘报告
7. 执行力识别与语气策略

其中最关键的是前 3 项，因为它们决定后面所有训练建议到底是“像教练”还是“像模板”。

## 9. P2 版本建议

P2 主要是差异化增强和商业化准备：

1. 伤病/异常风险专题
2. 多运动项目联合规划
3. 教练人格记忆
4. 月度阶段总结
5. 教练分享/协作视图

## 10. 最终发布顺序建议

如果只按投入产出比来排，推荐顺序如下：

1. 今日训练处方
2. 数据可信度提示
3. 今日身体状态卡重构
4. 每周训练负荷评估收口
5. 7 天基础训练计划
6. 历史分析可回看
7. Garmin 凭证安全改造
8. 个体化恢复评分模型
9. 自研训练负荷模型
10. 动态调计划

## 11. 一句话结论

首发不要试图一步做成“全能 AI 教练”，先把这件事做扎实：

**让用户每天打开后，清楚知道今天能不能练、该练什么、为什么这样练，以及未来 7 天怎么安排。**
