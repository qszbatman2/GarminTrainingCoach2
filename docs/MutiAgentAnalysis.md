# Garmin Multi-Agent 分析方案

## 1. 这份方案解决什么问题

目标只有一句话：

在尽量综合所有 Garmin 数据的前提下，给用户一个更准确、更稳定、而且更明确的训练指导结果。

这里的关键不是“让大模型看更多数据”，而是：

- 先把原始数据整理干净
- 再用规则层先定住底线判断
- 再让多个 agent 从不同角度补充分析
- 最后只输出一个统一行动建议

这样做的原因很简单：

- 只靠现在的单次分析模式，结果稳定，但解释维度不够多
- 只靠全量原始数据 + 大模型自由发挥，结果会更会说，但不够稳，也更贵

所以推荐采用混合模式：

`数据整理层 -> 规则判断层 -> 多 agent 分析层 -> 最终汇总层`

## 2. 用通俗的话理解整套系统

可以把整套系统想象成一个小型教练组：

- 数据整理层：数据整理员
- 规则判断层：裁判
- 恢复 agent：恢复教练
- 负荷 agent：训练负荷教练
- 目标 agent：目标管理教练
- 数据质量 agent：质检员
- 最终汇总器：总教练

每个人都只做自己那一部分事，最后由总教练给出一句统一结论。

用户最终不需要看到每个 agent 的原话，只需要看到：

- 今天能不能练
- 今天具体该怎么练
- 为什么这样安排
- 这一周接下来怎么调

## 3. 整体流程

```text
Garmin 原始数据
  ->
数据整理层
  ->
规则判断层
  ->
多 Agent 专项分析层
  ->
最终汇总层
  ->
前端展示一个统一结论
```

## 4. 每一步是谁在工作

## 4.1 第一步：数据整理层

### 这一步是谁在工作

数据整理器。

### 它在做什么

它负责把原始 Garmin 数据从“大而乱的 JSON”整理成“可分析的小字段”。

这一层不负责下结论，只负责把数据清洗干净。

### 它会处理哪些数据

- 睡眠
- HRV
- 静息心率
- 压力
- Body Battery
- 体重
- 训练记录
- 距离
- 时长
- 训练负荷
- 强度分钟
- 恢复时间
- 用户训练目标

### 它会做哪些事情

- 去掉明显无效的数据，比如最近几天同步失败造成的空白日
- 统一单位，比如秒转分钟、米转公里
- 拆出关键指标，比如最近 7 天、28 天、42 天的汇总结果
- 标记缺失字段，告诉后面系统“哪些地方不能说得太满”

### 这一步输出什么

- `cleanData`：清洗后的可用数据
- `timeWindows`：不同时间窗口的切片数据
- `dataQualitySummary`：缺失、异常、空白日说明

### 这一步的作用

保证后面的 agent 看的是“整理后的事实”，不是原始噪声。

## 4.2 第二步：规则判断层

### 这一步是谁在工作

规则引擎。

### 它在做什么

它负责先做硬判断，也就是先给出一个底线结论。

比如：

- 今天是否适合正式训练
- 当前恢复状态是不是明显异常
- 当前负荷是不是明显过高
- 这周训练量相对你过去一段时间到底偏低、合理，还是偏高

### 它会根据什么做判断

- 当日恢复指标
- 个人基线偏差
- 最近 7 天和 42 天负荷关系
- 最近一次训练刺激
- 本周训练量与近几周平均水平对比

### 这一步输出什么

- `decision`：`可训 / 慎训 / 不训`
- `riskFlags`：风险标签，比如恢复不足、负荷过高、连续休息过久
- `baselineSummary`：关键指标相对个人基线偏离多少
- `weeklyRuleSummary`：本周训练量和强度的规则结论

### 这一步的作用

这一层的意义是先把底线定住，避免后面的模型自由发挥把结论带偏。

## 4.3 第三步：恢复 agent

### 这一步是谁在工作

恢复教练。

### 它拿到什么信息

- 睡眠质量
- 睡眠时长
- HRV
- 静息心率
- 压力
- Body Battery
- 最近训练后的恢复时间

### 它在做什么

它只回答一个问题：

你今天的身体恢复状态，到底适不适合承受训练刺激。

它会重点判断：

- 你是恢复好了，还是明显没恢复
- 主要问题来自哪里
- 今天最大的风险点是什么

### 这一步输出什么

- `recoveryStatus`：恢复状态等级
- `recoveryReason`：恢复状态判断原因
- `recoveryAdvice`：从恢复视角出发的建议

### 示例

例如它可能会输出：

- 恢复一般
- 原因是 HRV 低于基线、睡眠评分偏低、最近一次训练恢复窗口未结束
- 建议今天不要安排高强度

## 4.4 第四步：负荷 agent

### 这一步是谁在工作

训练负荷教练。

### 它拿到什么信息

- 最近 7 天训练频次
- 最近 7 天训练时长
- 最近 7 天训练距离
- 训练负荷
- 强度分钟
- 近 4 周同进度平均值
- ATL / CTL 或对应代理值

### 它在做什么

它负责看你最近的训练节奏是不是合理。

它主要回答：

- 最近是不是练太少
- 最近是不是练太多
- 本周节奏和过去相比是偏低、合理还是偏高
- 当前问题是总量不够，还是强度不对

### 这一步输出什么

- `loadStatus`：训练负荷状态
- `loadReason`：为什么说偏低或偏高
- `loadAdvice`：从训练安排角度给出的建议

### 示例

例如它可能会输出：

- 本周总量偏低
- 与最近 4 周同进度相比，当前时长少了 30%
- 但高强度分钟没有明显不足
- 所以下一步重点不是补强度，而是补频次和稳定性

## 4.5 第五步：目标 agent

### 这一步是谁在工作

目标管理教练。

### 它拿到什么信息

- 用户训练目标
- 近期训练频次
- 近期训练结构
- 近期恢复状态
- 最近几周训练趋势

### 它在做什么

它负责判断你现在的训练方式，到底有没有朝目标前进。

它主要回答：

- 当前训练节奏和目标是否一致
- 如果不一致，差距在哪里
- 下一步最该补的是什么

### 这一步输出什么

- `goalAlignment`：当前是否对齐训练目标
- `goalGap`：与目标的核心差距
- `goalAdvice`：下一步为了更接近目标该怎么调

### 示例

例如用户目标是减重，它可能会输出：

- 当前和目标不一致
- 问题不在于单次训练强度不够，而在于每周训练频次不稳定
- 下一步要先把周频次拉回目标区间，而不是盲目加大单次训练量

## 4.6 第六步：数据质量 agent

### 这一步是谁在工作

数据质检员。

### 它拿到什么信息

- 空白日
- 缺失字段
- 异常值
- 同步失败痕迹
- 关键字段的可用程度

### 它在做什么

它不是帮你判断训练，而是帮系统判断“这次分析能信到什么程度”。

它会回答：

- 这次分析的可信度高不高
- 哪些结论可以直接说
- 哪些结论应该保守表达

### 这一步输出什么

- `confidenceLevel`：高 / 中 / 低
- `qualityWarnings`：数据质量警告

### 这一步的意义

避免系统把“数据没同步到”误说成“你真的应该休息”。

## 4.7 第七步：最终汇总层

### 这一步是谁在工作

最终汇总器，也就是总教练。

### 它拿到什么信息

- 规则引擎的底线结论
- 恢复 agent 的结论
- 负荷 agent 的结论
- 目标 agent 的结论
- 数据质量 agent 的结论

### 它在做什么

它负责做三件事：

- 检查不同 agent 之间有没有冲突
- 遇到冲突时，以规则引擎结论为底线
- 把多角度分析收敛成一个统一行动建议

### 这一步输出什么

- `finalDecision`：今天能不能练
- `finalAction`：今天具体该怎么做
- `finalReason`：为什么这么做
- `weeklyDirection`：这一周接下来怎么调
- `confidenceNote`：这次结论有多稳

### 这一步的作用

它是最后一道收口层，确保前端只看到一个明确答案，而不是四个专家各说各话。

## 5. 前端最终应该展示什么

前端不应该原样展示所有 agent 输出。

前端最终只展示 4 类信息就够了：

- `今天结论`
- `今天怎么做`
- `为什么`
- `本周怎么调`

推荐展示形式如下：

- 今天结论：可训 / 慎训 / 不训
- 今天怎么做：给一个明确动作方案
- 为什么：给一句到两句关键原因
- 本周怎么调：告诉用户接下来 3 到 7 天重点补什么

## 6. 一条理想的最终输出示例

```json
{
  "todayDecision": "慎训",
  "todayAction": "今天只做 30 到 40 分钟低强度有氧，不做间歇和冲量训练。",
  "why": "HRV 低于个人基线，睡眠评分偏低，最近一次训练后的恢复窗口还没有结束。虽然本周总量不高，但当前恢复信号不支持高强度刺激。",
  "weeklyDirection": "本周先补训练频次，不补强度。先连续完成 2 次轻松训练，再评估是否恢复到正常节奏。",
  "confidence": "中高，最近两天训练与恢复数据完整。"
}
```

## 7. 为什么推荐这种模式

### 相比现在的单次分析模式

优点：

- 能综合更多数据维度
- 能把恢复、负荷、目标、数据质量分开看
- 最终建议会更像教练，而不只是一个简单结论

缺点：

- 设计更复杂
- 需要更严格的输入切片和结果合并规则

### 相比全量原始数据直接交给大模型

优点：

- 稳定性更高
- 成本更可控
- 不容易出现模型乱推断
- 更适合长期回归和产品化

缺点：

- 不会像“纯自由分析”那样显得特别能说
- 需要工程上多做一层结构化整理

## 8. 最重要的设计原则

这套方案要成立，必须守住下面几条原则：

- 最终训练决策不能完全交给大模型自由发挥
- 所有数据都可以参与分析，但不能把所有原始 JSON 原样广播给每个 agent
- 每个 agent 只看自己需要的数据切片
- 最终只允许输出一个统一行动建议
- 如果数据质量差，系统必须明确告诉用户这次结论偏保守

## 9. 最终建议

如果目标是：

- 综合考虑所有数据
- 结果比现在更准确
- 输出更明确、更像教练
- 同时还要保持稳定和可控

那么最适合的方案不是“纯现在模式”，也不是“纯全量 LLM 自由分析”，而是：

`规则引擎主导底线决策 + 多 agent 分专题分析 + 最终汇总器统一输出`

这就是本方案推荐的 Multi-Agent Analysis 设计。
